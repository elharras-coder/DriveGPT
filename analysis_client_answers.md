## 💬 Ответы на встречные вопросы клиента

### Вопрос 1: Какие требования к хостингу?

Для сервиса, который запускает Python-ботов в Docker-контейнерах, нужен **VPS/VDS с root-доступом**, а не обычный shared-хостинг.

**Минимальные требования для 50 активных ботов:**

| Параметр | Минимум | Рекомендуется |
|----------|---------|---------------|
| CPU | 4 vCPU | 6-8 vCPU |
| RAM | 8 ГБ | 16 ГБ |
| Диск (SSD) | 80 ГБ | 120 ГБ |
| ОС | Ubuntu 22.04/24.04 | Ubuntu 24.04 |
| Docker | Обязательно | Docker + Docker Compose |
| Swap | 4 ГБ | 8 ГБ |

**Расчёт:** каждый бот — ~100-200 МБ RAM в контейнере, 50 ботов ≈ 5-10 ГБ RAM + система + база данных.

**Timeweb и Reg.ru:**
- **Timeweb Cloud VPS** — подойдёт. Есть тарифы с 8-16 ГБ RAM и Docker-поддержкой. Ориентировочно 2000-4000 ₽/мес.
- **Reg.ru VPS** — тоже подойдёт, но Timeweb Cloud обычно удобнее в управлении.

> [!IMPORTANT]
> Обычный shared-хостинг (виртуальный хостинг) **не подойдёт** — нужен именно VPS/VDS с root-доступом и возможностью установки Docker.

### Вопрос 8: Насколько сложно поддержать все Python-библиотеки?

**Несложно.** Если архитектура на Docker, то каждый бот запускается в своём контейнере с полноценным Python. Достаточно:

1. Поддержать `requirements.txt` в проекте бота
2. При сборке контейнера выполнять `pip install -r requirements.txt`

**Реализация:** +2-4 часа к базовой Docker-обёртке. **Рекомендация:** сделать сразу, это не добавит значительной сложности, а снимет ограничения.

### Вопрос 10: На что влияет размер проекта?

Размер проекта влияет на:
- **Время сборки Docker-образа** — больше файлов = дольше сборка
- **Дисковое пространство** — 100 проектов × 100 МБ = 10 ГБ только код
- **Время загрузки** — большие ZIP-архивы медленнее загружаются

**Рекомендация:** на старте можно не ограничивать, но поставить «мягкий» лимит в 50 МБ на проект с предупреждением. Для Telegram-ботов проекты обычно весят 1-5 МБ, так что проблем не будет.

**Несложно.** Базовая проверка — это просто вызов `python -m py_compile` для каждого `.py` файла.

```
Сложность: ~2-4 часа
Что даёт: отлавливает синтаксические ошибки ДО запуска
```

**Рекомендация:** включить в MVP — это займёт минимум времени, но сэкономит нервы при деплое. Ошибки типа «забыл закрыть скобку» будут пойманы сразу.

### Вопрос 15: Сложно ли реализовать предупреждения о дублях при импорте?

**Несложно.** Алгоритм:

1. Парсим CSV
2. Для каждой строки проверяем Telegram ID в базе
3. Если найден дубль — добавляем в список конфликтов
4. Показываем пользователю окно: «Найдено X дублей. Заменить / Пропустить / Отмена»

**Оценка:** +4-6 часов. **Рекомендация:** отложить на 2-ю итерацию. На старте загрузить базу один раз, дубли не возникнут.

**Да, это очень просто.** Буквально кнопка «Скачать CSV» + серверная генерация файла. 

**Оценка:** 2-3 часа. **Рекомендация:** можно включить в MVP — минимальные трудозатраты, полезный функционал.

### Вопрос 24: О чём позаботиться в плане безопасности?

Так как код пишете вы сами, риски значительно ниже. Но базово стоит обеспечить:

| Мера | Зачем | Приоритет |
|------|-------|-----------|
| Docker-изоляция | Бот не имеет доступа к хост-системе | 🔴 Критично |
| Лимиты ресурсов | Бот-с-багом не положит весь сервер | 🔴 Критично |
| Ограничение сети | Бот ходит только в Telegram API и Make | 🟡 Желательно |
| Бэкапы БД | Чтобы не потерять данные | 🔴 Критично |
| HTTPS на домене | Безопасная передача данных | 🔴 Критично |
| Firewall на VPS | Закрыть все порты кроме 80/443/22 | 🟡 Желательно |

**Рекомендация:** Docker-изоляция + лимиты ресурсов + HTTPS + бэкапы — это минимум, который нужно сделать сразу.

1. **Сжатые сроки** — 13 дней на полноценный сервис с Docker-инфраструктурой, это очень амбициозно
